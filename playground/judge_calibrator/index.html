Pick your Judge

L2 lab · the same AI answer, three different judges, three different scores.

Uncertainty dimension: variance around a score — the judge is an instrument with its own noise.

What this lab does

We take a small set of (question, AI answer, reference) triples — covering exact matches, paraphrases, partial answers, and wrong answers — and run three different judges over the same answers:

  • Strict — binary 0/1, exact match only
  • Lenient — binary 0/1, accepts paraphrases that contain the key information
  • Scale 1–5 — finer-grained ordinal scale, scores for partial credit

Each judge is an LLM call with a Jinja2 prompt template; in mock mode (no API key) a deterministic word-overlap heuristic stands in. The same answers pass under one judge and fail under another.

The single thing to feel: "accuracy" is not a property of the AI under test — it's a joint property of the AI and the judge. Change the rubric, change the model, change the temperature, and the same answers score differently.

Demo B — Re-run variance for the same judge

Now hold the judge fixed (Scale 1–5) and run it five times per example at temperature=0.7. With a non-zero temperature the LLM's output is no longer deterministic — same input, different score on different calls.

Requires LIVE=true in .env and an API key. With the mock fallback this demo trivially returns the same number each time.

What to notice: some examples lock in (Paris, the capital question — std 0). Others bounce across 3–5 on consecutive calls. That spread is the judge's own noise, not anything about the AI under test. If your eval scores each example once, your "accuracy" is one draw from this distribution.

The L2 takeaway

  1. The score depends on the judge. Same answers, different rubrics → different headline numbers. The judge is part of the measurement instrument.
  2. The score depends on the rubric's writing. Strict vs lenient is the same idea written differently — and the numbers diverge.
  3. The score depends on which re-run you happened to look at. Same judge, same example, non-zero temperature — and the score wobbles.

All three are the L2 dimension — variance around a score — the instrument has its own noise, distinct from L1's sampling noise (which comes from picking different cases).

Running it yourself

Open notebook.ipynb. pip install pandas python-dotenv jinja2 matplotlib, then run the cells. The lab works without API keys (mock fallback gives the cross-judge gap of Demo A). For Demo B copy .env.example to .env, fill in an API key, and set LIVE=true.
